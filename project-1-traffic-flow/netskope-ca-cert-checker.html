What's happening

The Netskope Client is steering HTTPS traffic to the NewEdge PoP. Netskope performs SSL inspection — it presents a certificate for salesforce.com that it dynamically signed using its own CA.

The problem: the device's OS trust store doesn't have the Netskope root CA installed. So when the browser checks the cert chain, it finds an issuer it doesn't recognize and throws NET::ERR_CERT_AUTHORITY_INVALID.

This happens for every single HTTPS site until the root CA is deployed.

Support call checklist — use this when a customer reports SSL certificate errors or "all HTTPS sites are broken" after deploying the Netskope client. Work through these in order.
0 of 12 checked 0%
1. Confirm the symptom
✓
Every HTTPS site shows cert error (not just one)
If only one site fails → likely cert pinning, not CA cert issue. If ALL HTTPS sites fail → CA cert not deployed.
✓
Error code is NET::ERR_CERT_AUTHORITY_INVALID (not ERR_CERT_DATE_INVALID)
Authority invalid = unknown CA. Date invalid = cert expired. Different problems, different fixes.
✓
Error only occurs when Netskope Client is running
Ask user to pause the client. If errors go away → confirms Netskope SSL inspection is the cause.
2. Check if CA cert is installed
✓
Mac: verify via Keychain Access
Open Keychain Access → System Roots → search "Netskope". Should show a trusted root CA entry. If missing → not deployed.
✓
Windows: verify via certmgr.msc
Run certmgr.msc → Trusted Root Certification Authorities → Certificates → look for Netskope. Or run certlm.msc for machine store.
✓
Check browser cert details — confirm issuer is "Netskope"
Click the broken padlock → Certificate → Issuer. If it says "Netskope" as issuer → CA is doing MITM but root not trusted. If issuer is something else → different issue.
3. Identify deployment method
✓
Is this a managed device (MDM-enrolled)?
Managed = deploy via Intune, JAMF, or GPO. Unmanaged/BYOD = user must install manually from Netskope tenant portal.
✓
Download the Netskope root CA from the tenant
In Netskope UI: Settings → Security Cloud Platform → Certificates → Download Root CA. This is the exact cert that needs to be deployed.
4. After deployment — verify fix
✓
Confirm cert is in OS trust store (not just browser)
Critical: must be in the OS store, not just imported into Chrome/Firefox. Native apps (Slack, Zoom, etc.) use the OS store, not the browser's.
✓
Restart browser after cert deployment
Browsers cache trust store state. A restart is required to pick up the newly installed root CA.
✓
Test HTTPS site — padlock should be green, issuer = Netskope
Click the padlock on any HTTPS site. Certificate issuer should now show "Netskope" and the chain should validate cleanly.
✓
Check SkopeIT events — SSL events should now appear for the user
If SSL inspection is working, you'll see page events and app events in SkopeIT for the user's traffic. If still blank → client not steering correctly.
1
Export the Netskope root CA from your tenant
In Netskope admin UI: Settings → Security Cloud Platform → Certificates → Root CA → Download. Save as netskope-root-ca.cer (DER or PEM format — Intune accepts both).
2
Create a Trusted Certificate profile in Intune
In Intune admin center → Devices → Configuration profiles → Create profile.

Platform: Windows 10 and later (or macOS)
Profile type: Templates → Trusted Certificate
Upload the netskope-root-ca.cer file.
Destination store: Computer certificate store → Root
⚠ Must be "Computer" store, not "User" store. Native apps and system processes use the computer store.
3
Assign to device groups
Assign the profile to the same device group that has the Netskope Client deployed. Include/Exclude groups can be used to scope to pilot users first before broad rollout.
✓ Tip: Deploy to a pilot group first and verify with one test device before rolling to all users. CA cert issues affect every HTTPS site — test carefully.
4
Verify deployment
On a target device after policy sync (dsregcmd /status or Settings → Accounts → Access work or school → Info → Sync):

# Windows PowerShell — verify cert in machine root store Get-ChildItem Cert:\LocalMachine\Root | Where-Object { $_.Issuer -like "*Netskope*" -or $_.Subject -like "*Netskope*" }
Should return the Netskope cert with a valid thumbprint. If empty → profile hasn't synced yet, force an Intune sync.
1
Export the Netskope root CA
Download from Netskope UI: Settings → Security Cloud Platform → Certificates → Root CA. Export as .pem or .cer format.
2
Create a Configuration Profile in JAMF Pro
Computers → Configuration Profiles → New

Add payload: Certificate
Upload the Netskope CA cert file.
System Keychain: Yes (this puts it in the System keychain, trusted for all users and apps).

⚠ "System" keychain vs "Login" keychain matters. System = all users + native apps. Login = only the current user's browser sessions.
3
Scope to target computers
Under Scope, target the computers or smart groups that have Netskope deployed. Click Save → the profile deploys on next JAMF check-in (or force: sudo jamf recon).
4
Verify on macOS
# Terminal — check System Keychain for Netskope cert security find-certificate -a -p /Library/Keychains/System.keychain \ | openssl x509 -noout -subject -issuer 2>/dev/null \ | grep -i netskope
Should return subject/issuer lines containing "Netskope". Also check Keychain Access app → System → Certificates → search "Netskope" → should show as trusted (blue+).
1
Export the Netskope root CA
Download from Netskope UI. Save as netskope-root-ca.cer in a network share accessible by all domain computers (e.g. \\domain\NETLOGON\certs\).
2
Create a GPO for certificate distribution
Open Group Policy Management Console (GPMC)
Create or edit a GPO linked to the target OU.

Navigate to:
Computer Configuration → Policies → Windows Settings → Security Settings → Public Key Policies → Trusted Root Certification Authorities

Right-click → Import → browse to netskope-root-ca.cer
⚠ Must be under Computer Configuration, not User Configuration, to apply to machine-level trust for all apps.
3
Force GPO update and verify
# Force GPO refresh on a test machine gpupdate /force # Verify cert installed in machine root store certutil -store Root | findstr /i "netskope"
✓ GPO cert distribution is reliable but can take up to 90 minutes to propagate to all machines via normal background refresh. Force update for immediate testing.
1
Download the cert from the Netskope portal
User navigates to their Netskope tenant portal or is provided the cert file by their admin. The cert is a standard .cer or .pem file — not sensitive, safe to distribute.
2
Install on macOS
Option A — Keychain (GUI): Double-click the cert file → Keychain Access opens → add to System keychain (not Login) → enter admin password → open Keychain Access → find the Netskope cert → double-click → expand Trust → set "Always Trust".

Option B — Terminal:
sudo security add-trusted-cert -d -r trustRoot \ -k /Library/Keychains/System.keychain \ netskope-root-ca.cer
⚠ Without the -d flag (add to admin certs) and -r trustRoot, the cert may install but not be fully trusted. Use both flags.
3
Install on Windows
Right-click the .cer file → Install Certificate → select Local Machine (not Current User) → Place all certificates in the following store → Browse → Trusted Root Certification Authorities → Finish.

# Or via PowerShell (run as Administrator) Import-Certificate -FilePath "netskope-root-ca.cer" ` -CertStoreLocation Cert:\LocalMachine\Root
4
Restart browser and verify
Close and reopen the browser. Visit any HTTPS site. Click the padlock → Certificate → check that the issuer chain now shows Netskope as the issuer and the cert is valid (green/trusted).
✓ If the user is on a managed device, they shouldn't need to do this manually — escalate to the admin to deploy via MDM/GPO instead.
