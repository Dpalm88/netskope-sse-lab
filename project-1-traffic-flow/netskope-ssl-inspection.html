SSL inspection failure modes — click to expand
📌
Certificate pinning
What happens: Some apps (banking apps, Zoom, some Google services) embed ("pin") the exact certificate or public key they expect from the server. When Netskope presents its own CA-signed cert instead, the app rejects the connection — even though the cert is technically valid.

Symptom: App works fine without Netskope client, breaks with it. Browser may work but native app fails.

Fix: Add the domain to the SSL bypass list in Netskope tenant settings. Traffic passes through uninspected — DLP and threat scanning blind for that domain, so only bypass what you must.
🔑
CA cert not deployed
What happens: Netskope presents its CA-signed cert to the client, but the client doesn't have the Netskope root CA in its trust store. Browser shows a scary NET::ERR_CERT_AUTHORITY_INVALID error for every HTTPS site.

Symptom: All HTTPS sites show SSL certificate errors immediately after installing the Netskope client.

Fix: Deploy the Netskope root CA certificate to managed devices via MDM (Intune, JAMF, GPO). Must be in the OS trust store, not just the browser — native apps use the OS store.
📋
Bypass list misconfiguration
Too broad: Bypassing entire domains like *.google.com means Gmail, Drive, Docs — all uninspected. DLP can't catch exfil to personal Google Drive. Huge policy gap.

Too narrow: Only bypassing mybank.com but not api.mybank.com or the CDN domains it depends on — app still breaks.

Fix: Use browser DevTools network tab to identify all domains an app contacts. Build precise bypass rules. Log bypass events in SkopeIT to audit what's escaping inspection.
🔒
TLS 1.3 + ECH (Encrypted Client Hello)
What happens: TLS 1.3 introduced Encrypted Client Hello (ECH), which encrypts the SNI (Server Name Indication) field — the part that tells Netskope which hostname the client is connecting to. Without SNI, Netskope can't categorize the destination, match policies by domain, or present the right cert.

Symptom: Sites using ECH may bypass category-based policies or show categorization errors in SkopeIT logs.

Fix: Netskope handles this via IP-based categorization fallback and DNS-over-HTTPS controls. Some tenants disable ECH at the network level (block DoH, force plaintext DNS) to maintain visibility.
🔐
Mutual TLS (mTLS)
What happens: mTLS requires the client to also present a certificate to the server (not just server → client). Netskope can't forward the client's private cert to the destination — it can only present its own identity. The destination rejects the connection.

Common scenario: API clients authenticating to internal services, IoT devices, some banking APIs.

Fix: Bypass mTLS-using domains in the SSL inspection policy. These flows can't be inspected inline — consider API-mode CASB scanning for data-at-rest coverage instead.
